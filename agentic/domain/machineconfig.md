# MachineConfig

**API Group**: `machineconfiguration.openshift.io/v1`  
**Kind**: `MachineConfig`  
**Scope**: Cluster

## Purpose

MachineConfig defines the desired operating system configuration for a set of machines. It uses CoreOS Ignition format to specify files, systemd units, kernel arguments, and OS updates.

**Key Principle**: MachineConfigs are immutable, static snapshots. The MCO merges multiple MachineConfigs into a "rendered" version for each pool.

## Spec Structure

```go
type MachineConfigSpec struct {
    Config          ign.Config  // Ignition v3 config
    KernelArguments []string    // Kernel boot arguments
    Extensions      []string    // RHCOS extensions to install
    Fips            bool        // Enable FIPS mode
    KernelType      string      // "default" or "realtime"
    OSImageURL      string      // Container image with OSTree payload
}
```

## Key Concepts

### Ignition Config

The `Config` field contains an Ignition v3 configuration that specifies:
- **Files**: Write files to `/etc` and `/var` (writable locations only)
- **Systemd Units**: Define services, timers, and their enablement
- **SSH Keys**: Configure `core` user SSH authorized keys
- **Directories**: Create directories (first boot only)

**Limitation**: Only files and systemd units are updatable in-place. Other Ignition sections cause degradation.

### Rendered MachineConfigs

MachineConfigs are merged into "rendered" configs (prefixed with `rendered-`):
- RenderController discovers all MachineConfigs matching a pool's selector
- Remote sources are fetched and embedded (no remote links in rendered config)
- Result is a static, complete machine configuration
- Named like `rendered-worker-a1b2c3d4` with content hash suffix

### OS Updates

The `OSImageURL` field points to a container image containing an OSTree payload:
- Controlled by cluster-version-operator via release image
- MCD uses `rpm-ostree` to apply OS updates
- Updates require reboot to activate new OS tree

### Kernel Arguments

Applied via bootloader configuration:
- Requires reboot to take effect
- Example: `["console=ttyS0", "nosmt"]`

### Extensions

RHCOS extensions are additional RPM content:
- Examples: `usbguard`, `kerberos`
- Installed via layered commits
- Requires reboot

## Ownership Model

MachineConfigs come from two sources:

1. **System-owned** (generated by TemplateController):
   - Prefix: `00-`, `01-`, `99-`
   - Generated from templates in `templates/`
   - Do not manually edit these
   - Examples: `00-master`, `01-worker-kubelet`

2. **User-created**:
   - Custom configurations for site-specific needs
   - Must have labels matching a MachineConfigPool selector
   - Examples: NTP configuration, custom systemd units

## Lifecycle

1. **Creation**: User or TemplateController creates MachineConfig
2. **Rendering**: RenderController merges configs for each pool → `rendered-*`
3. **Application**: UpdateController coordinates MCD to apply rendered config
4. **Validation**: MCD validates on-disk state matches config

## Example: Custom File

```yaml
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfig
metadata:
  name: 50-worker-custom-file
  labels:
    machineconfiguration.openshift.io/role: worker
spec:
  config:
    ignition:
      version: 3.2.0
    storage:
      files:
      - path: /etc/my-custom-config
        mode: 0644
        contents:
          source: data:,custom%20content
```

## Related Concepts

- **MachineConfigPool**: Groups machines and tracks rollout of rendered configs
- **KubeletConfig**: High-level API for kubelet configuration (generates MachineConfig)
- **ContainerRuntimeConfig**: High-level API for CRI-O configuration (generates MachineConfig)
